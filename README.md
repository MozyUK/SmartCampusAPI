### Part 1: Service Architecture & Setup
**Question: Explain the default lifecycle of a JAX-RS Resource class. Is a new instance instantiated for every incoming request, or does the runtime treat it as a singleton? Elaborate on how this architectural decision impacts the way you manage and synchronize your in-memory data structures (maps/lists) to prevent data loss or race conditions.**
In JAX-RS, the default lifecycle for a resource class is request-scoped. This means the runtime creates a new instance of the class for every single incoming HTTP request and discards it after the response is sent. Because the objects are ephemeral, instance-level variables cannot persist data. Therefore, our in-memory data structures (maps and lists) are declared as `static` to ensure they reside in the permanent class-level memory. To prevent race conditions in a multi-threaded environment where multiple requests might modify data simultaneously, we utilize thread-safe collections such as `ConcurrentHashMap` or synchronize access to the lists.

**Question: Why is the provision of ”Hypermedia” (links and navigation within responses) considered a hallmark of advanced RESTful design (HATEOAS)? How does this approach benefit client developers compared to static documentation?**
Hypermedia (HATEOAS) is considered an advanced RESTful pattern because it makes the API self-discoverable. Instead of forcing developers to hardcode URLs found in static documentation, the server provides the current available "actions" as links within the JSON response. This benefits client developers by decoupling the client from the server's URI structure; if a path changes on the server, the client—which simply follows the provided links—remains functional without needing code updates.

### Part 2: Room Management
**Question: When returning a list of rooms, what are the implications of returning only IDs versus returning the full room objects? Consider network bandwidth and client-side processing.**
Returning only IDs minimizes network bandwidth and reduces the initial payload size, making the response faster to transmit. However, it increases the processing burden on the client, as the client must then perform "N+1" additional requests to fetch the details for each specific ID. Returning full objects increases bandwidth usage per request but allows the client to render the full data immediately without further network latency.

**Question: Is the DELETE operation idempotent in your implementation? Provide a detailed justification by describing what happens if a client mistakenly sends the exact same DELETE request for a room multiple times.**
Yes, the DELETE operation is idempotent. Idempotency means that multiple identical requests have the same effect on the server state as a single request. In this implementation, the first DELETE request removes the room. Any subsequent identical requests will find that the room is already gone and return a 404 Not Found. Although the HTTP status code changes from the first request (204) to the subsequent ones (404), the state of the server (the room being deleted) remains exactly the same.

### Part 3: Sensor Operations & Linking
**Question: We explicitly use the @Consumes(MediaType.APPLICATION_JSON) annotation on the POST method. Explain the technical consequences if a client attempts to send data in a different format, such as text/plain or application/xml. How does JAX-RS handle this mismatch?**
If a client sends data in a format other than JSON, the JAX-RS runtime will detect the mismatch between the request's `Content-Type` header and the resource's `@Consumes` requirement. Consequently, the server will automatically reject the request and return an HTTP 415 Unsupported Media Type error. This ensures the application logic never attempts to process incompatible data formats.

**Question: You implemented this filtering using @QueryParam. Contrast this with an alternative design where the type is part of the URL path (e.g., /api/v1/sensors/type/CO2). Why is the query parameter approach generally considered superior for filtering and searching collections?**
Using `@QueryParam` is superior for filtering because query parameters are inherently optional and combinable. Using path parameters for filtering (e.g., `/sensors/type/CO2`) implies a rigid hierarchical resource location, which makes it difficult to add multiple filters (like type AND status) without creating complex and confusing URL patterns. Query parameters keep the URL focused on the resource collection while providing a flexible syntax for narrowing down results.

### Part 4: Deep Nesting with Sub-Resources
**Question: Discuss the architectural benefits of the Sub-Resource Locator pattern. How does delegating logic to separate classes help manage complexity in large APIs compared to defining every nested path (e.g., sensors/{id}/readings/{rid}) in one massive controller class?**
The Sub-Resource Locator pattern promotes the Single Responsibility Principle. By delegating nested paths to a separate class (like `SensorReadingResource`), the parent class remains clean and focused only on sensor-level operations. This separation makes the code more maintainable and readable, as developers do not have to navigate a single "God Class" containing hundreds of lines of code for every possible nested operation.

### Part 5: Advanced Error Handling, Exception Mapping & Logging
**Question: Why is HTTP 422 often considered more semantically accurate than a standard 404 when the issue is a missing reference inside a valid JSON payload?**
An HTTP 404 signifies that the URI itself was not found. When the URI is correct and the JSON payload is syntactically valid but contains a "broken" reference (like a non-existent `roomId`), the error is semantic rather than structural. HTTP 422 (Unprocessable Entity) explicitly indicates that the server understands the content type and the syntax of the request is correct, but it was unable to process the contained instructions due to logical errors.

**Question: From a cybersecurity standpoint, explain the risks associated with exposing internal Java stack traces to external API consumers. What specific information could an attacker gather from such a trace?**
Exposing stack traces provides an attacker with a roadmap of the application's internal architecture. It can reveal specific library versions (e.g., Jersey 3.1), class names, package structures, and even clues about the server's file system. An attacker can use this information to identify known vulnerabilities (CVEs) associated with those specific library versions or to understand how to bypass security logic by knowing exactly where an error occurs.

**Question: Why is it advantageous to use JAX-RS filters for cross-cutting concerns like logging, rather than manually inserting Logger.info() statements inside every single resource method?**
Using filters ensures centralized and consistent logic for cross-cutting concerns. Manual logging is prone to human error—a developer might forget to add a log in a new method—and clutters business logic. JAX-RS filters (ContainerRequestFilter and ContainerResponseFilter) automatically intercept every request and response, ensuring that all traffic is recorded uniformly, even if a request fails before reaching a resource method.
